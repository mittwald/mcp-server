### WP04: Auto-Generation Pipeline
**Duration**: Script development  
**Scope**: Build tool handler extraction and reference generation

**Deliverables**:
1. `docs/reference/scripts/extract-mcp-tools.ts`
   - Parses `/src/handlers/tools/` directory
   - Extracts: name, domain, parameters, return type, description
   - Output: `tools-manifest.json`

2. `docs/reference/scripts/generate-openapi.ts`
   - Converts `tools-manifest.json` → OpenAPI 3.0 schema
   - Output: `openapi.json`

3. `docs/reference/scripts/convert-to-markdown.ts`
   - Converts `openapi.json` → markdown files
   - Organizes by domain: `src/content/docs/tools/{domain}/{tool}.md`
   - Creates domain landing pages

4. `docs/reference/scripts/validate-coverage.ts`
   - Verifies 115 tools present
   - Checks for duplicates, missing fields
   - Output: `coverage-report.json`

5. Integration in `package.json`:
   ```json
   {
     "scripts": {
       "generate-references": "tsx scripts/extract-mcp-tools.ts && tsx scripts/generate-openapi.ts && tsx scripts/convert-to-markdown.ts",
       "validate-references": "tsx scripts/validate-coverage.ts",
       "prebuild": "npm run generate-references && npm run validate-references"
     }
   }
   ```

## Implementation Tips

### Divio Quality Checklist

**Tutorials** (Case Studies):
- ✅ Learning-oriented: New user can follow and complete
- ✅ Concrete: Real-world scenario with outcomes
- ✅ Beginner-friendly: No assumed expertise
- ✅ Complete: All steps present, no gaps
- ✅ Linked: References to reference docs and explainers

**How-To** (OAuth Guides):
- ✅ Goal-oriented: Clear problem/solution
- ✅ Practical: Step-by-step instructions
- ✅ Assumption: User has basic knowledge
- ✅ Flexible: Options and variations included
- ✅ Troubleshooting: Common problems addressed

**Reference** (Tool Docs):
- ✅ Information-oriented: Factual, complete
- ✅ Organized: By domain, alphabetically
- ✅ Consistent: Same format for all tools
- ✅ Examples: At least one per tool
- ✅ Links: Related tools and case studies

**Explanation** (Conceptual Docs):
- ✅ Understanding-oriented: "Why" before "what"
- ✅ Conceptual: Not instructional ("do this")
- ✅ Context: Historical/background information
- ✅ Diagrams: Visual support where helpful
- ✅ Design: Alternatives and trade-offs explained

### Write the Docs Best Practices

- ✅ **Accessibility**: Proper headings (H1 → H2 → H3), alt text for images, plain language
- ✅ **User-focused**: Written for audience, not internal
- ✅ **Versioned**: Docs live in git, reviewed before merge
- ✅ **Linked**: Cross-references between Divio types
- ✅ **Maintained**: Updated when features change

---

## Common Pitfalls to Avoid

❌ **Don't**: Write explanations that are really how-tos ("follow these steps")  
✅ **Do**: Explain concepts, design decisions, trade-offs

❌ **Don't**: Assume reader has prior knowledge in how-to guides  
✅ **Do**: Explain terms and concepts; link to explanation docs for deeper understanding

❌ **Don't**: Auto-generate tool reference pages with incomplete information  
✅ **Do**: Run validation script; ensure all fields populated; manual review of samples

❌ **Don't**: Forget alt text for diagrams and screenshots  
✅ **Do**: Every image has descriptive alt text for accessibility

❌ **Don't**: Create broken links between sites  
✅ **Do**: Test cross-site links; use correct relative paths; validate in build
