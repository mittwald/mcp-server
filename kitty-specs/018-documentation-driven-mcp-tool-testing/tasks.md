# Work Packages: Documentation-Driven MCP Tool Testing

**Inputs**: Design documents from `/kitty-specs/018-documentation-driven-mcp-tool-testing/`
**Prerequisites**: plan.md (architecture), spec.md (user stories), research.md (Pino logging, scenario schema), data-model.md (entities), contracts/ (JSON Schema), quickstart.md (usage guide)

**Tests**: Feature 014 regression testing required - ensure existing tool evals still pass after logging infrastructure changes.

**Organization**: 22 fine-grained subtasks (`T001`-`T022`) roll up into 6 work packages (`WP01`-`WP06`). Each work package is independently deliverable and testable. Work packages map to the 3 parallel streams identified in plan.md: Infrastructure (WP01-WP04), Scenarios (WP05), Reporting (WP06).

**Prompt Files**: Each work package references a matching prompt file in `/tasks/` generated by `/spec-kitty.tasks`. This file is the high-level checklist; deep implementation detail lives in prompt files.

## Subtask Format: `[Txxx] [P?] Description`
- **[P]** indicates the subtask can proceed in parallel (different files/components).
- Paths follow single project structure: `src/`, `evals/`, `tests/`

## Work Package WP01: MCP Logging Infrastructure (Priority: P0)

**Goal**: Implement production-quality structured logging with Pino for MCP tool call tracking and sanitization.
**Independent Test**: MCP server logs tool calls in structured JSON format with OAuth tokens redacted; Feature 014 evals still pass.
**Prompt**: `/tasks/WP01-mcp-logging-infrastructure.md`
**Estimated Size**: ~350 lines

### Included Subtasks
- [ ] T001 Install Pino dependencies and create Winston-to-Pino migration plan
- [ ] T002 Create structured logger with Pino config and sanitization utility
- [ ] T003 Add tool call instrumentation to MCP handlers
- [ ] T004 Validate logging with Feature 014 regression test

### Implementation Notes
1. Install `pino`, `pino-pretty`, `fast-redact` packages
2. Extract sanitization logic from `packages/oauth-bridge/src/middleware/request-logger.ts` to shared `src/utils/sanitize.ts`
3. Create `src/utils/structured-logger.ts` with Pino config (redaction paths, log levels)
4. Update `src/handlers/tool-handlers.ts` to log tool calls (start, success, error events)
5. Run Feature 014 evals (`npm run eval:run:create`) to verify no regressions

### Parallel Opportunities
- T002 (logger setup) and T003 (instrumentation) can be done in parallel if T001 completes first
- T004 (validation) must run sequentially after T001-T003

### Dependencies
- None (starting package, foundation for all subsequent work)

### Risks & Mitigations
- **Risk**: Logging overhead impacts tool call performance
  - **Mitigation**: Pino's async workers minimize overhead (<1ms per log); benchmark with T004
- **Risk**: Sensitive data leakage (OAuth tokens, API keys)
  - **Mitigation**: Comprehensive redaction paths + pre-sanitization function
- **Risk**: Breaking existing Feature 014 evals
  - **Mitigation**: T004 explicitly validates backward compatibility

**Independent Test**: After running 1 scenario, tool-validation.json is updated with validated tools; can query coverage status.
**Prompt**: `/tasks/WP03-coverage-tracking-tool-validation.md`

- [ ] T010 Implement coverage tracker (update ToolValidationRecord)

- **Risk**: Concurrent scenario runs overwrite coverage data
  - **Mitigation**: Use file locking or atomic writes for tool-validation.json

## Work Package WP04: Failure Pattern Analysis (Priority: P1)

**Goal**: Cluster failures by root cause (OAuth scope, timeout, not found) for actionable diagnostics.
**Independent Test**: After running scenarios with failures, failure-patterns.json contains clustered patterns with root causes.
**Prompt**: `/tasks/WP04-failure-pattern-analysis.md`
**Estimated Size**: ~350 lines

### Included Subtasks
- [ ] T012 Implement error signature extraction (normalize, classify)
- [ ] T013 Build failure clustering algorithm (token-based grouping)
- [ ] T014 Create failure pattern report generator

### Implementation Notes
1. Create `evals/scripts/failure-analyzer.ts` with:
   - `normalizeError()`: Replace UUIDs, IDs, numbers with placeholders
   - `extractSignature()`: Classify error_type from message patterns
   - `hashSignature()`: Generate pattern ID
2. Create `evals/scripts/cluster-failures.ts` to:
   - Read failed ToolValidationRecord entries
   - Extract signatures
   - Group by signature hash
   - Generate FailurePattern entries
3. Create `evals/scripts/generate-failure-report.ts` for Markdown output

### Parallel Opportunities
- T012 (extraction) and T013 (clustering) are sequential
- T014 (report generation) can be done in parallel after T013

### Dependencies
- **Depends on WP03**: Requires tool validation records with failure details

### Risks & Mitigations
- **Risk**: Too many unique patterns (low clustering effectiveness)
  - **Mitigation**: Normalize aggressively (resource IDs, timestamps, numbers)
- **Risk**: Pattern descriptions not actionable
  - **Mitigation**: Include recommended fixes in pattern templates

## Work Package WP06: Reporting & Gap Analysis (Priority: P2)

**Goal**: Generate JSON + Markdown coverage reports and identify uncovered tools for custom scenario creation.
**Independent Test**: Can generate coverage report showing validation rate (%) and failure patterns; gap analysis identifies uncovered tools.
**Prompt**: `/tasks/WP06-reporting-gap-analysis.md`
**Estimated Size**: ~380 lines

### Included Subtasks
- [ ] T019 [P] Build JSON coverage report generator
- [ ] T020 [P] Build Markdown coverage report generator
- [ ] T021 Implement gap analysis (identify uncovered tools)
- [ ] T022 Create retry-failures script

### Implementation Notes
1. Create `evals/scripts/generate-coverage-report.ts` to:
   - Read tool-validation.json (115 tools)
   - Calculate validation_rate, coverage_by_domain
   - Include failure pattern summary
   - Output JSON (`evals/reports/coverage-full.json`)
2. Create `evals/scripts/generate-markdown-report.ts` to:
   - Read coverage-full.json
   - Generate human-readable Markdown tables
   - Output to `evals/reports/coverage-summary.md`
3. Create `evals/scripts/gap-analysis.ts` to:
   - Compare tools used in scenarios vs 115-tool inventory
   - Identify uncovered tools
   - Output `evals/coverage/gap-analysis.json` with recommendations
4. Create `evals/scripts/retry-failures.ts` to re-run only failed scenarios

### Parallel Opportunities
- T019 (JSON report) and T020 (Markdown report) can proceed in full parallel
- T021 (gap analysis) depends on T019 (uses coverage report data)
- T022 (retry script) can proceed in parallel with T019-T021

### Dependencies
- **Depends on WP03**: Requires tool validation records
- **Depends on WP04**: Requires failure patterns for report summary
- **Depends on WP05**: Requires scenario execution results

### Risks & Mitigations
- **Risk**: Report generation is slow (blocking)
  - **Mitigation**: Cache tool inventory, use streaming for large files
- **Risk**: Gap analysis recommends too many custom scenarios
  - **Mitigation**: Prioritize by tool tier (Tier 0-2 first, Tier 4 last)
