# Implementation Plan: Real-World Use Case Functional Testing

**Branch**: `007-real-world-use` | **Date**: 2025-12-05 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/kitty-specs/007-real-world-use/spec.md`

## Summary

Build a comprehensive functional testing system that exercises all 170+ Mittwald MCP tools through 30-50 realistic multi-step use cases. The system extends the existing 005 harness with bidirectional streaming control, enabling a supervisory controller to detect completion states, enforce timeouts, and handle agent questions. Each use case is self-contained: create resources, perform work, gather evidence via Playwright, then cleanup.

## Technical Context

**Language/Version**: TypeScript 5.x (Node.js 20+)
**Primary Dependencies**: Claude CLI (`--input-format stream-json`), Playwright, existing 005 harness
**Storage**: JSONL session logs (file-based), JSON use case definitions
**Testing**: Vitest (unit), Playwright (E2E verification)
**Target Platform**: macOS/Linux CLI environment
**Project Type**: Single project extending tests/functional/
**Performance Goals**: Each use case completes within 30 minutes maximum
**Constraints**: Real Mittwald account (no mocks), must cleanup all created resources
**Scale/Scope**: 30-50 use cases achieving 100% tool coverage (170+ tools)

## Work Packages

### WP01: Bidirectional Streaming Infrastructure (P1)

**Goal**: Enable controller to inject responses into running Claude sessions

**Scope**:
- Extend SessionRunner to track stdin handle
- Create stdin writer utility for JSON message injection
- Implement message format: `{"type": "user", "message": {"role": "user", "content": "..."}}`
- Add test to verify round-trip communication

**Key Files**:
- `tests/functional/src/harness/session-runner.ts` (modify)
- `tests/functional/src/harness/stdin-injector.ts` (new)

**Acceptance**: Can inject a user message mid-session and observe Claude's response

**Dependencies**: None (foundational)

---

### WP02: Question Detection in Stream Parser (P1)

**Goal**: Detect when Claude asks questions requiring user input

**Scope**:
- Add question pattern detection to StreamParser
- Patterns: `?`, "Could you", "Would you like", "Please confirm", etc.
- Emit `question_detected` event with message content
- Track pending question state

**Key Files**:
- `tests/functional/src/harness/stream-parser.ts` (modify)

**Acceptance**: Parser emits events when assistant messages contain questions

**Dependencies**: None (foundational)

---

### WP03: Supervisory Controller (P1)

**Goal**: Orchestrate multi-step use cases with state awareness

**Scope**:
- Create SupervisoryController class
- Monitor execution state: pending, running, success, failure, timeout
- Detect "done" signals based on use-case-specific success patterns
- Detect "failure" patterns: repeated errors, stuck indicators
- Enforce configurable timeouts (default 10min, max 30min)
- Handle questions: answer from predefined responses, skip, or escalate

**Key Files**:
- `tests/functional/src/harness/supervisory-controller.ts` (new)

**Acceptance**: Controller correctly identifies success/failure/timeout for test scenarios

**Dependencies**: WP01 (stdin injection), WP02 (question detection)

---

### WP04: Use Case Type Definitions (P1)

**Goal**: Define data structures for use cases and executions

**Scope**:
- UseCase interface: title, description, prompt, expectedDomains, successCriteria, cleanupRequirements, estimatedDuration, toolCoverageMapping
- UseCaseExecution interface: startTime, endTime, status, sessionLogPath, evidenceArtifacts, cleanupStatus
- UseCaseCategory enum: apps, databases, domains-mail, etc.
- Validation schemas (Zod or similar)

**Key Files**:
- `tests/functional/src/use-cases/types.ts` (new)

**Acceptance**: Type definitions compile and can represent sample use cases

**Dependencies**: None (foundational)

## Risk Mitigation

| Risk | Mitigation |
|------|------------|
| Claude CLI stdin injection doesn't work | Fallback: Write to temp file, restart with new prompt |
| Mittwald rate limits during testing | Implement delays between use cases, batch by domain |
| Cleanup fails due to resource dependencies | Parse dependency order, multiple cleanup passes |
| DNS propagation delays | Polling with exponential backoff, configurable timeouts |
| 100% coverage not achievable | Document unreachable tools with justification |
